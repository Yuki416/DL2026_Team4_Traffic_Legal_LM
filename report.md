# GPT-4o 自動化資料集問題分析報告

## 報告目的

本專案以「司法院裁判書 → GPT-4o 結構化抽取 → 腳本白話化改寫 → Alpaca 格式」的流程，產出兩份微調資料集（二月、三月）。本報告記錄這兩份資料集中**由 GPT-4o 抽取環節與製作流程引入的品質問題**，作為專案成果報告的資料品質章節，並作為後續「精細處理版資料集」的改進依據。

本報告中所有統計數字均直接由資料檔案實測產生（非估計）。

## 二、二月份資料集問題（341 筆，全刑事）

### 2.1 資料分佈失衡

| 責任類型 | 出現筆數 | 佔比 |
|---|---|---|
| 違反道路交通安全注意義務 | 341 | **100%** |
| 刑事過失傷害責任 | 273 | 80% |
| 刑事過失致死責任 | 48 | 14% |
| 刑事過失致重傷責任 | 20 | 6% |
| **民事 / 侵權相關** | **0** | **0%** |

- **民法完全缺席（0/341）**：資料來源僅刑事判決，結構上不引用民法，導致責任類型完全無民事賠償面向。
- **單一 instruction**：341 筆共用同一句 instruction，模型容易學到「看到這句 → 輸出固定 header」的捷徑。

### 2.2 GPT-4o 抽取問題

| 問題 | 數據 | 說明 |
|---|---|---|
| 法條截斷到「條 / 項」 | 到款僅 6%（51/845） | 「道交規則第102條第1項」未抽到關鍵的「第7款」，喪失情境辨別力（102-1-7 轉彎讓直行 vs 102-1-2 左讓右，法律意義完全不同） |
| 核心爭議套語化 | 「相當因果關係」80%、「注意義務」100% | 法律套語過度泛化，情境不同的案子核心爭議高度雷同 |
| 法條粒度不足 | 平均每筆僅 2.5 條 | 抽取不完整，常漏掉判決實際引用的條文 |

### 2.3 白話化腳本問題

| 問題 | 數據 | 說明 |
|---|---|---|
| Input 結尾固定模板 | 53%（180/341） | 結尾固定為「以我這次…來說，會不會被認定要負…責任？」 |
| 情緒句型少數輪替 | — | 「壓力很大」「不知道該怎麼面對」等少數幾種句型輪替 |

## 三、三月份資料集問題（2,243 筆，刑事 + 民事）

三月份補進了民事資料，但 GPT-4o 抽取層與白話化腳本的同類問題依舊存在，並引入了**民事資料特有的新問題**與**判決書程序用語誤植**。

### 3.1 GPT-4o 抽取問題（與二月同類，未解決）

**問題 A：法條仍截斷到「條 / 項」，到款率反而更低（5%）**

| 精確程度 | 條目數 | 佔比 |
|---|---|---|
| 精確到「款」 | 608 | **5%** |
| 只到條 / 項 | 12,623 | 95% |

法條總條目 13,231 條，平均每筆 5.9 條（較二月的 2.5 條增加），但到款率從二月的 6% 降到 5%。**抽取量變多，精確度沒提升**。

**問題 B：核心爭議仍高度套語化**

| 核心爭議 | 出現筆數 | 佔比 |
|---|---|---|
| 駕駛過失與被害人受傷結果是否具有相當因果關係 | 1,054 | 47% |
| 被害人是否與有過失及雙方肇事責任比例 | 1,004 | 45% |
| 是否未注意車前狀況並隨時採取必要安全措施 | 905 | 40% |

「相當因果關係」「與有過失比例」「未注意車前狀況」三句通用法律套語覆蓋近半資料，與二月份同樣的泛化問題。

### 3.2 民事資料的新問題

**問題 C：民事責任類型 header 100% 制式化**

781 筆民事案件中：

| 民事責任類型 | 佔比 |
|---|---|
| 民事侵權行為損害賠償責任 | **100%（781/781）** |
| 汽機車駕駛人運行中損害賠償責任 | **100%（781/781）** |
| 車輛毀損修復費用賠償責任 | 75% |
| 人身損害賠償責任 | 57% |

前兩個 header 在**每一筆**民事案件都出現，等於沒有區辨力。模型只要看到「民事」instruction，就能無腦輸出這兩句，與二月份「違反道路交通安全注意義務 100%」是同一種制式化問題在民事面的翻版。

**問題 D：「告訴乃論案件程序責任」噪音（刑事中佔 46%）**

| 項目 | 筆數 | 佔比 |
|---|---|---|
| 責任類型含「告訴乃論案件程序責任」 | 672 | 全體 30%（刑事 1,462 筆中佔 **46%**） |
| 核心爭議含「告訴撤回 / 不受理」 | 655 | 全體 29% |

近半刑事案件因雙方和解、告訴撤回而以「不受理」結案。GPT-4o 把「告訴撤回 → 不受理」這個**程序結果**抽成核心爭議與責任類型。這類案件的判決理由通常只有幾句話、法條引用粗略，等於把大量「沒有實質過失認定」的低資訊案件當成正常訓練樣本，稀釋了真正有法律推理的案件。

### 3.3 白話化腳本問題（與二月同類 + 新增抽取錯誤）

**問題 E：Input 結尾仍是固定模板**

| 模板 | 佔比 |
|---|---|
| 刑事「以…來說，會不會被認定要負…責任？」 | 47%（1,061 筆） |
| 民事「像這種情況，對方要不要負民事損害賠償責任？」 | 35%（781 筆） |

兩種模板合計覆蓋 82% 的資料。民事的問句模板甚至是**100% 固定**（每一筆民事都用同一句結尾）。

**問題 F：判決書程序用語 / 法律用語被誤當成「事故地點」**

白話化腳本從判決書事實段抽取「地點」時，正則或欄位對應錯誤，把非地點文字塞進「在 X 附近」的句型：

| 誤植內容 | 出現筆數 | 性質 |
|---|---|---|
| 在「第一審辯論終結」附近 | 24 | 訴訟程序用語 |
| 在「事故發生後」附近 | 9 | 時間副詞 |
| 在「注意及此」附近 | 8 | 判決理由用語 |
| 在「同日17 / 同日12 / 同日下午6…」附近 | 多筆 | 時間被當地點 |
| 在「劃有分向限制線之路段」附近 | 6 | 法律描述 |

實際錯誤樣本（jid: SCDM,115,交易,1）：

> 事發時，我開著自小客車經過**第一審辯論終結附近**時，經過行人穿越道時沒有禮讓行人…

「第一審辯論終結」是訴訟程序用語，被腳本誤抽成事故地點。這是**製作流程的 bug**，不是 GPT-4o 的問題，但直接污染了 input 的真實性。

**問題 G：完全重複的 input**

2,243 筆中有 8 筆 input 與其他筆**完全相同**（唯一 input 數 2,235）。例如「事發時，我開車經過**快車道依規定駕車行駛**附近時…」出現 4 次——此處「快車道依規定駕車行駛」同樣是法律描述被誤當地點，且整句重複。

## 五、對訓練結果的實證影響

上述問題在 Round 1（二月資料，3 epochs，無 validation set）的微調結果中已得到實證（詳見 `eval/Round1_3 AI Case Evaluation.md`）：

| 觀察到的模型行為 | 對應資料集問題 |
|---|---|
| 三題測試「責任類型」一字不差（純刑事，無民事） | D1（民法缺席）+ G3（header 制式化） |
| 「是否左轉時未禮讓直行車」被硬套到無左轉的情境 | G2（核心爭議套語化）+ P1（input 模板化） |
| 「相當因果關係」三題硬套 | G2 |
| 法條只到條未到款，關鍵法條（93-1-2）缺失 | G1 |
| loss 收斂到 0.145 是記憶而非泛化 | P1 + P4（捷徑學習）+ 訓練設定 |

**核心診斷：fine-tuning 治格式、不治內容。** 模型 100% 學會了 JSON 格式（資料集的優點），但同時忠實複製了資料集的所有缺陷（模板、套語、偏刑事、法條不精）。

## 六、結論與改進方向

### 6.1 三月份相較二月份的進步

- ✅ 補進民事資料（35%），解決民法完全缺席的最嚴重缺口
- ✅ Instruction 區分刑事 / 民事
- ✅ 資料量擴大 6.5 倍（341 → 2,243）

### 6.2 仍待解決的核心問題

GPT-4o 抽取層的三大系統性問題（G1 法條不精、G2 核心爭議套語化、G3 header 制式化）在三月份**並未改善**，且新增了 G4（程序噪音）與 P2（地點誤植）。這說明：**單純擴大資料量、增加月份，無法修正抽取品質本身的問題。**

### 6.3 對「精細處理版資料集」的具體建議

| 優先 | 改進項目 | 對應問題 |
|---|---|---|
| 🔴 高 | 法條從判決書**主文/理由原文**抽取，精確到「款」，不用 GPT-4o 推斷 | G1 |
| 🔴 高 | 核心爭議改為**個案特異**描述，加入品質驗證步驟過濾套語 | G2 |
| 🔴 高 | 責任類型 header 去除「逢案必填」的通用項，保留有區辨力的內容 | G3 |
| 🟡 中 | 過濾「告訴撤回 / 不受理」的低資訊程序性案件，或單獨標記 | G4 |
| 🟡 中 | 重寫白話化腳本的地點抽取邏輯，修正程序用語誤植 | P2 |
| 🟡 中 | Input 問句模板多樣化，打破固定結尾 | P1 |
| 🟢 低 | 去除完全重複的 input | P3 |

### 6.4 評估方法建議

二月、三月份資料的「正確性」目前無法量化（核心爭議 / 責任類型來自 GPT-4o，本身就是待驗證對象）。後續應以三月份判決書的**主文與理由原文**作為 Ground Truth（法官實際認定，可查證），建立 Precision / Recall / F1 的量化評估，才能把「套語複讀程度」轉成可追蹤的指標。

## 七、新版規則式 Pipeline（Step 1-4）改善實作與驗證

本章記錄第六章「改進方向」在新版三月資料集中的**實際實作方式**與**實測結果**，逐一對應 G1-G4、P1-P4、D1。新版資料 = `data/colloquial/training_pairs.json`（1,759 筆，刑事 783 / 民事 976），來自全新流程：

```
司法院裁判書開放資料（202603，105,384 筆）
        ↓ Step 1：規則式篩選（刑事四層 + 民事三層條件，見 3月份資料集.md）
  traffic_cases_merged.json（2,159 筆）
        ↓ Step 2：人工 QC（30 樣本抽查）
        ↓ Step 3：結構化抽取（regex/關鍵字，零 LLM，見 如何結構化抽取判決書.md）
  structured_cases.json（1,918 筆：事故摘要 / 核心爭議 / 責任類型 / 適用法條）
        ↓ Step 4：白話化改寫（Sonnet 4.6 sub-agent，32 chunk 並行，見 docs/step4_workflow_record.md）
  training_pairs.json（1,759 筆 Alpaca 訓練對）
        ↓ Step 5：切分
  train.json(1,679) / val.json(50) / test.json(30)
```

**與 GPT-4o 流程的根本差異**：Step 1（篩選）與 Step 3（結構化抽取，含核心爭議、責任類型、適用法條）**完全不經過 LLM**，全部是規則式 / regex / 關鍵字比對，直接從判決書原文（JFULL）抽取。LLM（Sonnet 4.6）只負責 Step 4 的「事故摘要 → 口語改寫」這一個語言轉換任務，且其輸出（input 文字、grounded_disputes 標籤）會再經過接地驗證與 sanitizer 過濾。**G 系列問題（GPT-4o 抽取層的系統性缺陷）因為抽取主體不再是 LLM 而結構性消失**；P 系列問題則由 Step 4 的明確規則與多樣化要求解決。

---

### 7.1 G1：法條截斷到條/項，極少到款 → regex 直抓原文 + 不可學法條過濾

**機制**：

1. Step 3 用 regex `第(\d+(?:-\d+)?\s*條(?:第\d+項)?(?:第\d+款)?(?:前段|後段|但書)?)`，**只抓 JFULL 原文中實際出現的文字**——原文寫到「款」就抓到「款」，不靠 LLM 推斷補齊。
2. Step 3 另做 span 重疊過濾，移除「保險法」被「強制汽車責任保險法」字串包含而產生的幽靈法條（詳見 如何結構化抽取判決書.md 第 4 節）。
3. Step 4 組裝時再過濾**量刑/程序類條文**（刑法第 41/50/51/53/55/57/71/74/75 條、民法第 203/229/233 條）——這些條文（緩刑、數罪併罰、遲延利息等）無法從事故描述本身推論，保留只會讓模型學到「看到某情境就背出固定法條」的捷徑記憶。

**實測數字**：

| 指標 | 二月（GPT-4o） | 三月 GPT-4o | **新版（Step1-4）** |
|---|---|---|---|
| 平均每筆法條數 | 2.5 條 | 5.9 條 | **2.97 條** |
| 精確到「款」 | 6%（51/845） | 5%（608/13,231） | **12.8%（669/5,224）** |

到款率提升至 GPT-4o 版本的 2 倍以上。法條總數從 5.9 條降到 2.97 條，並非抽取量減少，而是**過濾掉了 14.2%（1,011/7,125）的量刑/程序性條文**——這些條文在原始抽取階段確實存在（regex 抓到原文），但在白話化的 input 中無對應線索，因此不放入訓練輸出。

> 「143 筆法條只有 1 條」逐筆核對為小額簡易判決原文確實只引一條（如「民法第184條第1項前段」），**屬實反映原文，非抽取失敗**（如何結構化抽取判決書.md 已逐筆驗證）。

**狀態**：✅ 已解。

### 7.2 G2：核心爭議是「通用法律問句」而非「案件情境描述」→ 改為 12 類情境標籤 + Step4 語意接地

**GPT-4o 版本實際長相**：「核心爭議」欄位是 GPT-4o 生成的完整句子，三月版前三名如下：

| 核心爭議（GPT-4o 原文，逐字）| 出現筆數 | 佔比 |
|---|---|---|
| 駕駛過失與被害人受傷結果是否具有相當因果關係 | 1,054 | 47% |
| 被害人是否與有過失及雙方肇事責任比例 | 1,004 | 45% |
| 是否未注意車前狀況並隨時採取必要安全措施 | 905 | 40% |

問題不在「同一句話出現很多次」本身，而在於：**這三句話是「過失責任認定」的通用法律提問框架，句子本身不包含任何能區分「這是轉彎未讓、闖紅燈、還是支線道未讓幹線道」的資訊**。一起「路口闖紅燈相撞」的案件和一起「變換車道擦撞」的案件，核心爭議欄位完全可能是同一句「駕駛過失與被害人受傷結果是否具有相當因果關係」——這句話對「模型該學會辨認哪種情境」沒有提供區分依據。

**新版機制**：

1. Step 3 的 `match_disputes()` 改用 **12 類情境標籤**（如「轉彎車未讓直行車」「支線道未讓幹線道」「違反號誌」「變換車道未讓直行車」），對 JFULL 全文做 regex/關鍵字直接比對命中——標籤本身就是「事故動態的分類」，不是法律問句。0 筆落入清單外的「其他」。
2. Step 4 改寫完 input 後，要求 sub-agent 做「**grounded_disputes（語意接地）**」判斷：標籤若來自判決書「理由」段的法律推理、但改寫後的事故描述本身完全沒提到對應情境，則拿掉；反之若改寫後文字能合理支撐某個清單內標籤（即使原始核心爭議沒列），則可以新增。確保**每一個標籤都能在 input 裡找到對應的具體描述**。

**狀態**：✅ 已解。

### 7.3 G3：責任類型 header 制式化 → 改為類別標籤，非生成式句子

**機制**：

新版的「責任類型」**從設計上就不是生成式句子**，而是固定詞彙的類別標籤集合：

```python
def determine_responsibility(record):
    if record["_source"] == "criminal":
        if "賠償" in jfull and "損害賠償" in jfull:
            return ["刑事", "民事"]
        return ["刑事"]
    else:
        return ["民事"]
```

輸出永遠是「刑事」「民事」「刑事、民事」三者之一，不存在「民事侵權行為損害賠償責任」「汽機車駕駛人運行中損害賠償責任」這類**逢民事案必填的長句 header**。

**實測數字（最終 1,759 筆）**：

| 責任類型 | 筆數 | 佔比 |
|---|---|---|
| 民事 | 976 | 55.5% |
| 刑事 | 737 | 41.9% |
| 刑事、民事 | 46 | 2.6% |

GPT-4o 三月版的問題是「民事侵權行為損害賠償責任」「汽機車駕駛人運行中損害賠償責任」**100% 出現在每一筆民事案件**，等於零區辨力。新版因為輸出 schema 本身是簡短類別標籤而非生成句子，**這類「逢案必填的制式 header」在資料結構上不可能出現**——不是事後過濾掉，而是欄位設計階段就不存在這個自由度。

**狀態**：✅ 已解（結構性消除，非後製過濾）。

### 7.4 G4：程序結果被當成實質爭議 → Step 1 篩選階段排除整案

**機制**：

Step 1 刑事 pipeline 第三層「整案駁回排除」：

```python
FULL_DISMISS_PATTERNS = ["本件公訴不受理", "本案公訴不受理"]
CONVICTION_PATTERNS   = ["處有期徒刑", "處拘役", "處罰金"]
```

### 7.6 P2：地點抽取誤植程序/法律用語 → 整段語意改寫取代欄位式地點抽取

**機制**：

**實測數字**：對最終 1,759 筆 input 掃描「第一審辯論終結」「辯論終結」「注意及此」「分向限制線之路段」「事故發生後附近」等 GPT-4o 版本曾出現的誤植詞彙，**殘留 0 筆**。

**兩種設計取捨，而非單純「模板化 vs 不模板化」**：

- gpt4o：「標準要件層」確實是逐案套用的法律課本句式，但**疊加在上面的個案特異層內容更豐富**（涵蓋酒駕、闖紅燈左轉、僱用人連帶責任、車損折舊估價等 sonnet46 schema 完全沒有的面向）——資訊量明顯較大
- sonnet46：封閉詞彙（12 類），每案只選 2-3 個最相關的標籤，10/12 類被實際用到，dispute_f1 = 0.811，顯示確實依個案差異化選擇，而非固定套用同一組標籤

**但「資訊量大」不等於「可驗證」**：sonnet46 的標籤是封閉詞彙，可以直接算 precision/recall/F1（已知 dispute_f1=0.811）；gpt4o 個案特異層的 26 種自由句子**沒有對應的封閉 ground truth 可比對**——句子讀起來言之有物，但無法量化「這句話是否真的對應到該案的法院認定」，可信度需要人工逐筆檢查才能確認。

### 8.6 三個具體案例：實際輸出長什麼樣

**案例一｜CDEV,114,橋小,1304（無照駕駛肇事，保險代位求償）**

> 使用者描述：「...保險公司說那個司機沒有駕照，他們要跟他求償...對方無照開車我可以另外再請求嗎？」

| | sonnet46 | gpt4o |
|---|---|---|
| 適用法條 F1 | 0.36 | 0.67 |
| 抓到「無照駕駛」對應法條？ | ❌ 沒有 | ✅ 道路交通管理處罰條例第21條 |

→ sonnet46 完全沒提到「無照駕駛」，因為**我們的 12 類核心爭議標籤裡沒有「無照駕駛」這一類**——模型沒有「掛鉤」把這個事實連到對應法條。這是 **schema 設計的盲點**，不是模型訓練失敗。

**案例二｜TPEV,115,北小,151（sonnet46 最差案例，law_f1 = 0）**

| | gold | sonnet46 預測 |
|---|---|---|
| 適用法條 | 民法第125條（**消滅時效**）| 民法第184/196/213條（典型「過失賠償」三件套）|

→ 這個案件的核心爭議其實是「時效抗辯」，不是常見的「誰有過失」問題。sonnet46 遇到**非典型案件**時，會退回最常見的「過失賠償」pattern，完全錯過真正爭點。

**案例三｜CDEV,114,橋小,54（law_f1 = 0.33，方向對但條號錯）**

| | gold | sonnet46 預測 |
|---|---|---|
| 適用法條 | 道交安規第**99**/**91**條、民法184條 | 道交安規第**94**條、民法184/195條 |

→ 大方向（道路交通安全規則 + 民法184）抓對了，但**規則內的具體條號搞混**（94 vs 99/91）——相近條號是模型的弱點。

### 8.7 結論：兩種路線的取捨

| | sonnet46（新版資料集）| gpt4o（教學組三月資料集）|
|---|---|---|
| 輸出格式 | ✅ 簡潔三行，好解析、推論快 | ⚠️ 冗長 JSON，長度約 2-3 倍 |
| 核心爭議 | ✅ 12類封閉標籤，依個案選2-3個（F1 0.811，可驗證）| ⚠️ 平均6.2句，前8句近乎逐案套用「標準要件」，其餘26種個案特異但無 ground truth 可驗證對錯 |
| 適用法條範圍 | ⚠️ 經過濾、較窄（平均3.7條）| ✅ 較廣（平均7.1條），F1 數字較高 |
| 適用法條盲點 | ⚠️ 12類schema有缺口（無照駕駛、消滅時效等無對應類別）| ✅ 自由生成不受類別限制，能抓到周邊法條 |
| 法條噪音 | ✅ 過濾掉不可學的量刑/程序條文 | ⚠️ 含大量「逢案必填」樣板條文（民法229/233等），貢獻57%的TP |

**一句話**：sonnet46 走「精簡、結構化、低噪音、可驗證」路線，12 類 schema 仍有覆蓋盲點；gpt4o 走「廣覆蓋、自由生成」路線，資訊量大、F1 數字（0.701）較好看，但代價是法條清單夾雜大量樣板噪音、核心爭議的個案特異內容缺乏可驗證的 ground truth。**下一步建議擴充 12 類 schema**（補上無照駕駛、消滅時效等類別）並修正相近條號混淆，吸收 gpt4o 個案特異層的覆蓋廣度，同時保留封閉詞彙的可驗證性。
