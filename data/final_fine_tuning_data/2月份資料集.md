## 一、資料組成

### 來源

- **原始資料**：二月份台灣交通事故刑事判決書，從刑事法院資料庫篩選
- **篩選條件**：三層過濾（JCASE 子字串 → JTITLE → PATH_KEYWORDS in JFULL）
- **資料量**：341 筆

### 製作流程

```
原始判決書（JFULL 全文）
  → Step 3：GPT-4o 結構化抽取（核心爭議 / 責任類型 / 適用法條）
  → Step 4：腳本白話化改寫（生成口語問題作為 input）
  → Alpaca 格式（instruction / input / output）
```

### Alpaca 格式欄位

| 欄位 | 內容 | 說明 |
|---|---|---|
| `instruction` | 請判斷以下車禍情境的核心爭議、責任類型與適用法條 | 全部 341 筆完全相同 |
| `input` | 口語化事故描述 | 由腳本將判決書事實段改寫而成 |
| `output` | JSON 字串 | 含三個欄位：核心爭議、責任類型、適用法條 |
| `jid` | 判決書識別碼 | 可回溯至原始判決書 |

### Output JSON 結構

```json
{
  "核心爭議": ["...", "..."],
  "責任類型": ["...", "..."],
  "適用法條": ["...", "..."]
}
```

全部 341 筆 output 均可解析為合法 JSON。

## 三、資料集描述

本資料集為 Round 1 LoRA 微調訓練集，以台灣二月份交通事故刑事判決書為原始素材，透過 GPT-4o 結構化抽取與腳本白話化改寫製成。

**資料集定位：**
- 任務：接收口語化事故描述，輸出法律結構化分析（核心爭議 / 責任類型 / 適用法條）
- 適用模型：Llama-3-Taiwan-8B-Instruct + LoRA
- 使用輪次：Round 1 Training Set（全數用於訓練，無 Validation Set）

**已知侷限：**
- 資料來源僅涵蓋刑事判決，不含民事損害賠償判決
- 全部 341 筆共用同一 instruction，輸出多樣性受限
- 訓練結果（loss = 0.1451）顯示模型高度擬合此資料集

## 四、發現的資料集問題

問題依歸因方向分為三類。

---

### 4.1 GPT-4o 處理不夠好的部分

**問題 G1：法條抽取精度不足（94% 缺款）**

GPT-4o 從判決書中抽取法條時，系統性地截斷到「條」或「項」，沒有保留到「款」。例如輸出「道路交通安全規則第102條第1項」，而非「第102條第1項**第7款**」。不同款的法律意義完全不同（102-1-7 是「轉彎讓直行」，102-1-2 是「左讓右」），缺款使法條喪失情境辨別力。

**問題 G2：核心爭議套語化，情境區分度不足**

「駕駛過失與被害人受傷結果是否具有相當因果關係」出現 273/341 筆（80%），「違反道路交通安全注意義務」出現 341/341 筆（100%）。這些法律套語在判決書理由段普遍存在，GPT-4o 在抽取時過度泛化，沒有根據個案的核心衝突調整。結果是情境不同的案子，核心爭議欄高度雷同。

**問題 G3：法條幻覺（32%）**

約 32% 的法條是 GPT-4o 推斷補入，並非原始判決書引用的條文。這些推斷在法律邏輯上不一定錯，但無法對應到法官實際採用的依據，污染了輸出的正確性。

---

### 4.2 原始判決書本身的問題

**問題 J1：資料來源全為刑事判決，民法完全缺席**

從刑事法院資料庫篩選的判決書，結構上不引用民法（刑民分離）。導致資料集中民法條文 0/341（0%），責任類型完全無民事賠償面向。這是資料來源的結構性偏差，GPT-4o 無從彌補。

**問題 J2：情境分布集中，少見情境訓練不足**

左轉未讓直行（31%）與右轉（13%）佔多數，反映現實事故分布。但閃光號誌（3%）、無號誌路口（13%）等情境比例偏低，模型學習機會嚴重不足，導致遇到這些情境時套用多數情境的模板。

**問題 J3：部分判決書法條引用本身粗略**

輕傷和解案件的判決理由往往只有幾句話，法條引用本來就只到條或項。GPT-4o 沒有精確款項可以抄，只能輸出粗略版本，加劇了問題 G1。

---

### 4.3 資料集製作流程與任務設計的問題

**問題 P1：Input 生成腳本模板固定（53% 重複結尾）**

白話化腳本使用固定句型，造成 180/341 筆（53%）的 input 結尾有「以我這次⋯來說，會不會被認定⋯」這個模板。情緒句型也只有少數幾種輪替。模型學到的「口語問題」是高度人工化的語言，泛化到真實使用者的多樣問法時能力有限。

**問題 P2：Output Schema 未設計民事欄位**

三欄設計（核心爭議 / 責任類型 / 適用法條）沒有強制要求民事賠償分析。即使換上更豐富的原始資料，只要 schema 不改，民事面向就不會出現在輸出中。

**問題 P3：全部 341 筆共用同一 Instruction**

固定 instruction 讓模型更容易學到「看到這句話 → 輸出固定 header + 高頻詞彙」的捷徑，加速模板化。多樣化 instruction（如「民事賠償怎麼算？」、「我需要負刑事責任嗎？」）會迫使模型對應到不同輸出方向。

**問題 P4：資料庫選擇只涵蓋刑事，未納入民事判決**

只從刑事資料庫篩選是流程設計的決策，而非判決書本身的問題。若製作流程同時納入民事損害賠償判決書，問題 J1 就不會發生。

**問題 P5：製作流程缺乏品質驗證步驟**

GPT-4o 生成後沒有自動或人工的品質過濾：法條幻覺（G3）、法條截斷（G1）、模板套語（G2）全部直接進入訓練集。若加入「對照原始判決書逐筆驗證法條是否吻合」的步驟，G1 和 G3 在進訓練前就能被過濾。

**問題 P6：訓練設定放大了資料集缺陷**

3 epochs + 無 Validation Set + 無 early stopping，讓模型對資料集的缺陷反覆記憶。loss 收斂到 0.1451 是過擬合的訊號，而非泛化能力的體現。資料集本來就有模板問題，跑越多輪，模板越固化。
